# UX/UI аудит дашборда и аналитики пользователей (2026-04-18)

## Контекст и охват

Этот аудит выполнен как **экспертный UX/UI и продуктовый review по коду интерфейсов** (без юзабилити-тестов с респондентами).

Охват:
- Основной Dashboard v2 (layout, sidebar, navigation, вкладки/секции).
- Экран аналитики страницы (`InsightsScreen`) и CRM analytics panel.
- Бизнес-зона: `ZoneDashboard` и `ZoneAnalyticsScreen`.
- Админ-аналитика (`AdminAnalyticsDashboard`) как эталон/контраст по аналитическим паттернам.

---

## Executive summary

### Общая оценка UX/UI дашборда: **7.2 / 10**

Сильные стороны:
- Современная структура layout с desktop sidebar + mobile bottom nav и ясной информационной архитектурой.
- Богатая аналитическая функциональность: периоды, графики, top-content, экспорт в Excel/PDF, фильтры.
- Хороший baseline по micro-interactions (motion, skeleton, empty/error states).

Ключевые проблемы:
1. **Низкая консистентность терминов и периодов** между разными аналитическими поверхностями.
2. **Смешение языков (RU/EN) и неунифицированный microcopy**, что ухудшает доверие и скорость чтения.
3. **Разночтения в определениях метрик** (CTR/Conversion/Visitors), что снижает управленческую ценность.
4. **Потенциальные проблемы доступности**: очень мелкие подписи осей/легенд и reliance на цвет.

### Отдельная оценка аналитики пользователей: **6.5 / 10**

Основной риск — **качество данных и их интерпретируемость**:
- fallback-оценки для уникальных посетителей,
- разнобой ключей `visitorId/visitor_id`, `sessionId/session_id`,
- «All» период фактически не all-time.

## 1) Аудит UX/UI всего дашборда

## 1.1 Сильные стороны

1. **Адаптивная архитектура layout**
   - Desktop: боковая навигация с секциями.
   - Mobile: нижняя навигация.
   - Это снижает навигационную нагрузку и соответствует паттернам SaaS-продуктов.

2. **Хорошая модульность навигации**
   - Есть разделение на main items, секции и отдельный business block.
   - Визуально активная вкладка достаточно заметна.

3. **Наличие системных состояний**
   - Loading / Empty / Error присутствуют на аналитических экранах.
   - Это снижает “silent failure” и улучшает perception надежности.

4. **Сильный функционал аналитических экранов**
   - KPI + time series + breakdown + top content + экспорт.

## 1.2 Ключевые UX/UI проблемы (с приоритетом)

### P0–P1: Консистентность периодов и терминов

**Наблюдение:**
- В разных экранах используются разные наборы периодов:
  - `Today/Week/Month/All` (CRM Analytics).
  - `7d/14d/30d` (InsightsScreen).
  - `7d/30d/90d/all` (ZoneDashboard/ZoneAnalytics).
  - `7d/14d/30d/90d` (Admin).

**Почему это плохо:**
- Пользователь не формирует стабильную ментальную модель.
- Нельзя быстро сравнивать данные между модулями.

**Рекомендация:**
- Ввести единый стандарт периодов для всех analytics surfaces:
  - `7d`, `30d`, `90d`, `all-time`.
- Если нужен `today`, показывать как быстрый фильтр (secondary), а не основной стандарт.

### P1: Смешение языков и микрокопи

**Наблюдение:**
- В русскоязычных интерфейсах встречаются англоязычные метки (`Pipeline`, `Win Rate`, `Test CTR`, `Analytics & Reports`).

**Риск:**
- Снижение воспринимаемого качества и доверия.
- Более сложная сканируемость интерфейса.

**Рекомендация:**
- Утвердить единый glossary для RU (и отдельно EN).
- Прогнать consistency pass для всех лейблов аналитики.

### P1: Навигационная неоднозначность структуры dashboard

**Наблюдение:**
- В sidebar одновременно присутствуют `home`, `insights`, `zone-dashboard`, отдельные CRM/zone блоки.
- Для пользователя без onboarding это может ощущаться как “дублирующие дашборды”.

**Рекомендация:**
- Явно разделить уровни:
  - **Page analytics** (про контент страницы),
  - **Business analytics** (про сделки/финансы/операции).
- Добавить короткие подсказки под заголовками (1 строка “что вы здесь видите”).

### P2: Доступность (A11y) и читаемость

**Наблюдение:**
- Часто используются подписи осей 9–10px, мелкие secondary labels.
- Тренды и статусы сильно завязаны на цвет.

**Рекомендация:**
- Минимум 12px для осей на мобильном, 11px для dense desktop.
- Дублировать цвет семантическим текстом/иконкой (+/- и “рост/снижение”).

### P2: Потенциальная визуальная перегруженность

**Наблюдение:**
- На некоторых экранах много графиков и карточек в одном viewport.

**Рекомендация:**
- Добавить progressive disclosure:
  - 3–4 ключевых KPI сверху,
  - остальное в collapsible секциях “Подробнее”.

## 2) Отдельная аналитика пользователей (data + UX)

## 2.1 Сильные стороны

1. Широкий набор метрик: просмотры, клики, шеры, источники, устройства, гео, топ-блоки.
2. Есть сравнение с предыдущим периодом (trend).
3. Есть сегментация по staff/контенту и экспорт.

## 2.2 Критические проблемы качества пользовательской аналитики

### A1 (критично): Недетерминированность метрики unique visitors

**Наблюдение:**
- При отсутствии `visitorId` используется fallback `Math.ceil(totalViews * 0.7)`.

**Риск:**
- Число посетителей становится оценочным и может вводить в заблуждение.
- Продуктовые решения могут приниматься на неверной базе.

**Рекомендация:**
- Показывать `N/A` или “insufficient tracking data”, если visitorId не собирается стабильно.
- Ввести data quality badge (coverage % по visitorId/sessionId).

### A1 (критично): Несогласованность ключей metadata

**Наблюдение:**
- В коде используются и camelCase, и snake_case ключи (`visitorId`/`visitor_id`, `sessionId`/`session_id`).

**Риск:**
- Потеря части событий в агрегатах, нестабильные цифры между экранами.

**Рекомендация:**
- Контракт аналитики v1:
  - единый нейминг,
  - валидатор schema при записи события,
  - миграция historical events при необходимости.

### A2 (высокий): Семантическая путаница метрик

**Наблюдение:**
- “Conversion Rate” в админке по факту = clicks/views и подписан как “Test CTR”.
- В некоторых местах CTR и conversion трактуются смешанно.

**Риск:**
- Ложная интерпретация эффективности воронки.

**Рекомендация:**
- Развести определения:
  - CTR = clicks/views,
  - Conversion rate = conversions/visitors (или sessions) по событию-конверсии.
- В UI дать tooltip с формулой каждой метрики.

### A2 (высокий): Период `all` не является real all-time

**Наблюдение:**
- Для `all` в page analytics берется последние 12 месяцев.

**Риск:**
- Пользователь думает, что смотрит всю историю, а видит подмножество.

**Рекомендация:**
- Либо переименовать в “12 months”,
- либо реализовать true all-time + async/aggregated query.

### A3 (средний): Недостаток прозрачности источников и погрешности

**Наблюдение:**
- В UI не отражается степень полноты данных (сколько событий с device/source/visitor).

**Рекомендация:**
- Добавить data quality panel:
  - coverage по обязательным полям,
  - доля unknown,
  - задержка обновления.

## 3) Приоритизированный план улучшений

### Sprint 1 (быстрые победы, 1–2 недели)
1. Унифицировать периоды на всех analytics экранах.
2. Привести microcopy к одному языку (RU/EN отдельно) и словарю терминов.
3. Исправить названия метрик (CTR vs Conversion) + tooltip формулы.
4. Увеличить min размер подписей графиков и улучшить контраст.

### Sprint 2 (качество данных, 2–3 недели)
1. Ввести единый metadata schema contract для аналитики.
2. Удалить fallback `0.7 * views` для visitors, заменить на прозрачный `N/A`/coverage.
3. Добавить мониторинг data quality (coverage/unknown rates).

### Sprint 3 (product maturity, 3–5 недель)
1. Реализовать all-time агрегаты (или честный label “12 months”).
2. Добавить сравнение периодов side-by-side и cohort/retention view.
3. Добавить роль-ориентированные пресеты dashboard (creator / sales / admin).

---

## 4) KPI успеха после внедрения

1. **UX consistency score** (внутренний чеклист) ≥ 90%.
2. Снижение времени “до первого инсайта” на analytics экране минимум на 20%.
3. Рост доверия к данным (опрос) на +15 п.п.
4. Coverage visitor/session/device/source в событиях ≥ 95%.
