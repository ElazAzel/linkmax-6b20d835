# PAGE BUILDER AUDIT REPORT
**Date:** 2026-04-18  
**Scope:** Конструктор страниц (создание страницы + редактирование блоков)  
**Components reviewed:** `CreatePageDialog`, `BlockEditorV2`, `BlockManager`

## Методика аудита
- Ручной code review критического пути: создание страницы, открытие/закрытие редактора блока, сортировка структуры блоков.
- Анализ edge-cases: сетевые ошибки, повторное открытие модалок/дроверов, консистентность локального состояния.
- Проверка regression-рисков через существующие unit-тесты редактора и диалогов.

## Выявленные проблемы

### 1) CreatePageDialog: неустойчивость к rejected Promise при создании страницы
**Симптом:** если `onCreatePage()` завершался исключением (reject), диалог не показывал понятную ошибку и мог остаться в неконсистентном loading-состоянии.  
**Риск:** пользователь не получает корректный feedback, повторное создание страницы затрудняется.

**Исправление:**
- Обернул создание страницы в `try/catch/finally`.
- Гарантировал корректный сброс `loading` в `finally`.
- Добавил fallback-сообщение об ошибке для непредусмотренных исключений.

### 2) CreatePageDialog: слабая превентивная валидация перед submit
**Симптом:** кнопку можно было нажать даже при заведомо невалидном slug или пустом title, что приводило к лишним попыткам submit и ошибкам только постфактум.  
**Риск:** лишние запросы, ухудшение UX.

**Исправление:**
- Добавил вычисляемую валидацию slug через `useMemo`.
- Визуально отображаю ошибку под полем slug до отправки.
- Заблокировал кнопку `Create`, пока форма невалидна или идет загрузка.

### 3) BlockEditorV2: stale-state при повторном открытии того же блока
**Симптом:** при повторном открытии редактора того же блока (с тем же `id`) могла подтягиваться устаревшая локальная форма из предыдущей сессии, потому что синхронизация шла только при смене `id`.  
**Риск:** пользователь видит неактуальные данные и может перезаписать корректные данные устаревшей копией.

**Исправление:**
- Добавил синхронизацию `formData` при каждом открытии (`isOpen=true`), даже если `id` не изменился.
- Сбрасываю `hasUnsavedChanges` и `lastSaved` при открытии, чтобы статус редактора отражал текущее состояние.
- Уточнил типизацию формы (`Partial<Block>`) вместо `any`.

### 4) BlockManager: рассинхрон локального списка блоков при внешних изменениях
**Симптом:** если список блоков обновлялся извне, пока manager открыт, локальный `localBlocks` мог расходиться с source of truth.  
**Риск:** пользователь сортирует устаревший список, вероятность конфликтов порядка.

**Исправление:**
- Добавил `useEffect`, который при открытом менеджере синхронизирует `localBlocks` от `blocks`.
- Одновременно сбрасываю `hasChanges`, чтобы не показывать ложный dirty-state после внешнего апдейта.

## Результат
- Улучшена стабильность сценария создания страниц при сетевых и неожиданных ошибках.
- Снижен риск ошибочных submit-операций через упреждающую валидацию.
- Исправлен классический stale-state баг в редакторе блоков.
- Повышена консистентность структуры блоков при параллельных изменениях.
