# Стратегический план развития lnkmx (2026)

## 1. Резюме

lnkmx успешно завершила фазу **"Business OS Foundation"** (Q1 2026). Платформа теперь обладает полноценным CRM-ядром (сделки, задачи, инвойсы, ЭДО). Следующий этап развития сфокусирован на **эксплуатационной надежности (Reliability)**, **мобильном превосходстве (Mobile First)** и **реальной коммерциализации (Monetization)**.

---

## 2. Приоритетные направления (P0-P2)

### P0: Стабилизация и Инженерное Качество (Март - Апрель)

* **Устранение рассинхронизации хуков**: Проведение глобального аудита `src/components/zones/` для обеспечения обязательной передачи `zoneId` во все сервисные хуки.
* **E2E Тестирование**: Внедрение Playwright для критических путей Business Zones (создание сделки -> выставление счета -> генерация акта).
* **AI Decoupling**: Завершение отделения логики генерации Gemini от UI для возможности легкой замены моделей и повышения отказоустойчивости.

### P1: Мобильная Экосистема (Май - Июнь)

* **Native app (Capacitor)**: Переход от PWA к нативному приложению для сторов.
* **Push-уведомления**: Мгновенные алерты о новых лидах, оплатах инвойсов и дедлайнах по задачам.
* **Офлайн-режим**: Возможность работы с CRM без интернета с последующей синхронизацией.

### P1: Аналитика и Рост (Июль - Август)

* **Phase 4 Analytics**: Визуальные воронки продаж внутри зон. Анализ конверсии из лида в закрытый документ.
* **Automations V2**: Расширение триггеров автоматизации (например, "если задача просрочена на 24ч -> отправить уведомление в Telegram").

### P2: Финтех и Экосистема (Сентябрь - Декабрь)

* **Kaspi API / Robokassa**: Прямая оплата инвойсов клиентами пользователя. Автоматическая смена статуса счета на "Paid".
* **Public API (v1)**: Открытие API для интеграции с внешними инструментами (Zapier, Tilda, 1C).
* **White-Label Agency**: Модель для маркетинговых агентств — создание брендированных кабинетов для своих клиентов.

## 3. План реализации (Ближайшие шаги)

### Шаг 1: Технический аудит и фикс хуков

* Провести grep-поиск по `useZoneDeals`, `useZoneContacts`, `useZoneTasks`.
* Убедиться, что каждый вызов получает `zoneId`.
* Добавить в `TECH_DEBT_BACKLOG.md` прогресс по исправлению.

### Шаг 2: Проектирование воронки аналитики

* Спроектировать SQL-view для сбора метрик по зонам (время жизни сделки, средний чек, загрузка сотрудников).

### Шаг 3: Подготовка Capacitor-инфраструктуры

* Инициализация Capacitor в проекте. Настройка базового Splash Screen и иконок.

## 4. Риски и Митигация

* **Риск**: Увеличение сложности Business Zones может отпугнуть новых пользователей.
  * *Решение*: Сохранение "умного скрытия" (Progressive Disclosure) — расширенные функции доступны только при активации Business Zone.
* **Риск**: Вариативность API платежных систем (Kaspi/Robokassa).
  * *Решение*: Создание абстракного `PaymentProviderService` для легкой настройки провайдеров.
